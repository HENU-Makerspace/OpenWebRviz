# WebBot-Viz 潜在 Bug 审计报告

审计时间：2026-04-28

审计范围：从根目录 `bun dev` 入口开始，沿 `server`、`client`、ROS/导航控制、媒体链路、云端部署配置逐文件阅读。未纳入第三方 vendored `janus-gateway/src` 源码本体。

## 1. 启动链路概览

根目录 `package.json`：

- `bun dev` -> `bun run dev:local`
- `dev:local` -> `ROBOT_CONFIG_PROFILE=local concurrently "bun run dev:server:local" "bun run dev:client"`
- `dev:server:local` -> `ROBOT_CONFIG_PROFILE=local bun run --filter server dev`
- `dev:client` -> `bun run --filter client dev`

注意：`packages/server/package.json` 自己的 `dev` 默认是 `dev:cloud`。根目录脚本显式覆盖为 local，所以从根目录运行是本地配置；但如果开发者进入 `packages/server` 直接执行 `bun run dev`，会默认使用 cloud 配置。

## 2. 高优先级问题

### 2.1 导航模式没有使用“已选地图”的静态地图数据

证据：

- `App.tsx` 把 `selectedMap` 传入 `MapCanvas`，但 `MapCanvas` 内部没有使用它读取 `/api/maps/:name/data`。
- `MapCanvas.tsx` 仍始终订阅 `/map`：`useRosMap(ros, mapTopic, mapPaused)`。
- 导航模式的“冻结地图”逻辑只是冻结进入导航模式后的第一帧 `/map`，不是加载侧栏选中的地图。

影响：

- 用户在导航页选择地图后，画布显示的可能不是选择的地图，而是当前 ROS `/map` 发布的第一帧。
- 如果导航启动后 ROS 端尚未发布 `/map`，画布会一直显示“正在等待地图数据...”，即使云服务器已有保存地图。
- 如果选择 A 地图启动导航，但 `/map` 首帧来自 B 地图或旧状态，点击目标点会在错误底图上取坐标。

位置：

- `packages/client/src/App.tsx:777-788`
- `packages/client/src/components/MapCanvas.tsx:72-91`
- `packages/client/src/components/MapCanvas.tsx:95-104`

建议：

- 在 `MapCanvas` 中监听 `mode === 'navigation' && selectedMap`，调用 `/api/maps/${encodeURIComponent(selectedMap)}/data` 加载静态地图。
- 导航模式下优先显示该静态地图；实时 `/map` 可作为可选图层或仅用于未选地图时 fallback。
- `selectedMap` 变化时清空 `frozenNavMap`、路径和已选点，避免跨地图残留。

### 2.2 地图名称可路径穿越，上传/读取/删除都可能越过 `maps` 目录

证据：

- `/api/maps/upload` 直接使用 `name` 拼接路径。
- `/api/maps/:name/data` 和 DELETE 也直接拼接 `mapName`。

影响：

- 恶意或误操作请求可写入、读取、删除 `maps` 目录外的 `.yaml/.pgm` 目标。
- 当前 CORS 是 `origin: '*'`，接口又没有鉴权，云端部署时风险更高。

位置：

- `packages/server/src/index.ts:315-319`
- `packages/server/src/index.ts:579-581`
- `packages/server/src/index.ts:673-679`
- `packages/server/src/index.ts:696-704`

建议：

- 限制地图名为安全白名单，例如 `/^[a-zA-Z0-9_-]+$/`。
- 所有文件操作使用 `path.resolve(MAPS_DIR, filename)` 后检查结果必须以 `path.resolve(MAPS_DIR) + path.sep` 开头。
- 上传接口增加来源认证，至少加共享 token 或内网反代限制。

### 2.3 `192.168.1.58` 是当前改代码/直连调试机器人，需要和 cloud IoT 配置角色分开

证据：

- 用户确认 `192.168.1.58` 是当前用于改代码、免密登录、直连核对的机器人。
- `robot_config.local.yaml` 的 `jetson.host` 是 `192.168.1.58`。
- `robot_config.cloud.yaml` 的 `jetson.host` 是 `192.168.43.100`。
- `tools/ops-status.ts` 在 cloud profile 下会 SSH 到 `config.jetson.host`。
- 本次已只读核对：`nvidia@192.168.1.58` 可免密登录；`/home/nvidia/bin/webbot-*` 和本仓库 `packages/server/systemd/webbot-*` 模板 hash 一致。

影响：

- 这不应被简单判定为“机器人 IP 配错”；它更像是同一仓库同时服务两种路径：本地直连 `192.168.1.58`，云端 IoT/隧道配置 `192.168.43.100`。
- 真正的风险是文档和运维工具语义不清：`ops:status:cloud` 会按 cloud profile SSH `192.168.43.100`，而当前可改代码的目标是 `192.168.1.58`。
- 如果后续需要用工具直接核对当前机器人，应使用 local profile，或新增显式 `ops:status:robot` 指向 `192.168.1.58`。

位置：

- `packages/server/config/robot_config.local.yaml`
- `packages/server/config/robot_config.cloud.yaml`
- `tools/ops-status.ts`

建议：

- 文档里明确区分“本地开发机器人地址”和“云端 IoT/隧道侧地址”。
- 运维脚本拆分 `jetson_host_for_ssh` 与 `tunnel_target_host`，避免一个 `jetson.host` 同时承担两个含义。
- 保留 cloud 配置前，确认 `192.168.43.100` 是云端生产链路真实地址；否则 `ops:status:cloud` 可能误导。

### 2.4 `/api/network` 在 ESM 中使用 `require`

证据：

- `packages/server/package.json` 设置 `"type": "module"`。
- `index.ts` 顶部没有导入 `os`，但 `/api/network` 内部调用 `require('os').hostname()`。

影响：

- Bun 构建可通过，但运行到 `/api/network` 时可能触发 `require is not defined` 或平台相关异常。
- `system_manager_node.py` 的旧发现逻辑会请求 `/api/network`，如果该接口异常，会影响自动发现。

位置：

- `packages/server/src/index.ts:751-754`

建议：

- 改为 `import * as os from 'node:os';`，使用 `os.hostname()`。

### 2.5 PGM 解析过于脆弱，遇到标准注释行会错读地图

证据：

- PGM header 查找固定数 3 个换行。
- 解析维度时只过滤空行和 `P` 开头行，没有处理 `#` 注释。

影响：

- `map_saver_cli` 生成的 PGM 常见格式可能包含注释行，例如 `# CREATOR: ...`。
- 当前解析可能把注释行当成尺寸行，导致 `width/height/maxVal` 为 `NaN`，返回错误地图数据。

位置：

- `packages/server/src/index.ts:610-626`
- `packages/server/src/index.ts:628-656`

建议：

- 按 PGM token 规范解析：跳过空白和 `#` 到行尾注释，依次读取 magic、width、height、maxVal，再计算二进制数据偏移。
- 校验 `width * height === imageData.length`，否则返回 422。

## 3. 中优先级问题

### 3.1 保存地图上传失败仍返回成功

证据：

- `system_manager_node.py` 中地图本地保存成功后，上传失败只 `warn`，最终仍 `response.success = True`。
- 前端 `handleSaveMap` 固定等待 3 秒后刷新云服务器地图列表。

影响：

- 机器人本地保存成功但云服务器没有收到地图时，UI 仍可能表现为保存完成。
- 导航侧栏依赖云服务器 `/api/maps`，用户随后找不到刚保存的地图。

位置：

- `packages/server/src/system_manager_node.py:324-348`
- `packages/client/src/App.tsx:127-139`

建议：

- 将“本地保存成功”和“上传云端成功”拆分到响应 message，或上传失败时返回 `success=false`。
- 前端不要固定 sleep 3 秒；可以轮询 `/api/maps` 查找新地图，超时后展示明确错误。

### 3.2 前端启动导航后即使失败也清掉 loading，缺少错误展示

证据：

- `NavigationPanel.startNavigation` 调用 `await startNav(...)` 后直接 `setStarting(false)`。
- `useSystemManager.startNavigation` 捕获异常并只写入内部 `status.error`，没有 throw。
- 侧栏没有展示 `robotStatus.error`。

影响：

- 地图文件不存在、Nav2 params 不存在、服务类型不匹配等失败只停留在 console 或内部状态，操作者看不到失败原因。

位置：

- `packages/client/src/App.tsx:260-271`
- `packages/client/src/hooks/useSystemManager.ts:116-128`

建议：

- `NavigationPanel` 展示 `robotStatus.error`。
- `startSlam/startNavigation/stopAll/saveMap` 返回结构化结果，UI 根据结果显示成功/失败。
- `try/finally` 确保 `starting` 在异常路径也会恢复。

### 3.3 手动“重新连接”可能被旧 `connectionStateRef` 阻挡

证据：

- `reconnect()` 中先 `setConnectionState('disconnected')`，随后立即 `connect()`。
- `connect()` 判断的是 `connectionStateRef.current`；该 ref 只在 React effect 中同步，可能仍是 `connected` 或 `connecting`。

影响：

- 用户点击重连时可能没有实际发起新连接，需要等待状态 effect 或再次点击。

位置：

- `packages/client/src/hooks/useRosConnection.ts:79-82`
- `packages/client/src/hooks/useRosConnection.ts:209-216`

建议：

- 在 `reconnect()` 里同步写 `connectionStateRef.current = 'disconnected'` 后再 `connect()`。
- 或让 `connect({ force: true })` 跳过当前状态保护。

### 3.4 导航动作完成判断可能把“失败/取消”当成功

证据：

- `useNavigationTasks` 使用 `sendGoal` 的第二个回调作为成功路径，错误回调作为失败路径。
- 对 ROS2 action 来说，最终 result 里可能需要检查 status/result，而不是仅靠回调触发。

影响：

- Nav2 goal 被 rejected、aborted、canceled 时，UI 可能误报“已完成”，尤其在 roslib 行为和后端 action 状态映射不一致时。

位置：

- `packages/client/src/hooks/useNavigationTasks.ts:174-240`

建议：

- 明确 roslib v2 `Action.sendGoal` 回调签名，读取 result/status。
- 对 canceled/aborted/rejected 做独立状态显示。
- 增加真实 action mock 测试。

### 3.5 多点巡航是前端串行单点，不使用 `/navigate_through_poses`

证据：

- `/api/config` 下发了 `navigateThroughPosesAction`。
- `useNavigationTasks` 没有创建 `navigate_through_poses` action，只反复调用 `NavigateToPose`。

影响：

- Nav2 无法整体规划多点路线，前端刷新、断线或页面关闭会丢失队列。
- 循环巡航完全依赖浏览器页面常驻。

位置：

- `packages/server/src/index.ts:365-370`
- `packages/client/src/hooks/useNavigationTasks.ts:93-105`
- `packages/client/src/hooks/useNavigationTasks.ts:189-214`

建议：

- 途经多点使用 `NavigateThroughPoses`。
- 循环巡航最好放到机器人端或后端任务管理，前端只下发/取消任务。

### 3.6 导航点位朝向在屏幕预览和 ROS 坐标计算上方向可能不一致

证据：

- 发送 pose 时 `theta = atan2(endWorldY - startWorldY, endWorldX - startWorldX)`。
- 画箭头时使用屏幕坐标 `atan2(end.y - start.y, end.x - start.x)`，而屏幕 Y 轴向下。
- 机器人箭头绘制又使用 `arrowAngle = -displayPose.theta`。

影响：

- 拖拽箭头显示方向可能与实际发给 Nav2 的 yaw 相反或存在 Y 轴翻转差异，操作者容易以为目标朝向已经设置正确。

位置：

- `packages/client/src/components/MapCanvas.tsx:185-187`
- `packages/client/src/components/MapCanvas.tsx:404-409`
- `packages/client/src/components/MapCanvas.tsx:439-447`
- `packages/client/src/components/MapCanvas.tsx:519-536`

建议：

- 预览箭头统一用 world theta 计算后再 `worldToScreen` 两端点。
- 增加一个已知姿态的视觉/单元测试：向右、向上、向左、向下拖拽分别对应 0、pi/2、pi、-pi/2。

### 3.7 媒体运行时依赖外部 CDN

证据：

- WebRTC adapter 固定从 cdnjs 加载。

影响：

- 云服务器内网、现场网络或安全策略禁止访问外部 CDN 时，视频/对讲功能无法初始化。
- 当前后端已经对 `adapter_asset` 返回 404，配置含义和运行时行为不一致。

位置：

- `packages/client/src/hooks/useRobotMedia.ts:69`
- `packages/client/src/hooks/useRobotMedia.ts:130-132`
- `packages/server/src/index.ts:389-390`

建议：

- 将 adapter 打进前端 bundle，或由后端 `/api/media/assets/adapter.min.js` 提供本地文件。
- 删除不再使用的 `adapter_asset` 配置，或让配置真正生效。

### 3.8 Talkback forwarder 的目标地址在云端隧道场景下语义不清

证据：

- cloud/local 配置中 `audiobridge_forward_host` 都是 `127.0.0.1`。
- 后端运行在云服务器时，`forwardJanusRequest` 通过隧道访问 Jetson Janus；Janus 插件里的 `127.0.0.1` 是相对 Janus 所在机器，也就是 Jetson。

影响：

- 如果 `webbot-media-control.py` 或音频回放链不在 Jetson 本机监听 `127.0.0.1:5006`，对讲 RTP 会发错位置。
- 这个配置在本地和云端看起来一样，但实际参照物是 Janus 主机，不是后端主机，容易误配。

位置：

- `packages/server/config/robot_config.cloud.yaml`
- `packages/server/src/index.ts:38-39`
- `packages/server/src/index.ts:508-519`

建议：

- 文档中明确 `audiobridge_forward_host` 是“Janus 视角的目标地址”。
- 在 `/api/media/status` 中增加回放端口健康探测。

## 4. 低优先级/可维护性问题

### 4.1 简易 YAML parser 容易误解析

证据：

- `parseYamlConfig` 只支持一级 section + 简单 key。
- 带小数点的值全部作为字符串保存，再由调用方 `Number(...)` 补救。
- 不支持数组、嵌套、行内注释、带 `:` 的复杂字符串。

影响：

- 当前配置大多能工作，但一旦配置文件增加嵌套或复杂值，会静默丢字段或变成字符串。

位置：

- `packages/server/src/config.ts:10-44`

建议：

- 使用正式 YAML 解析库，例如 `yaml`。
- 对配置做 schema 校验，启动时输出缺失字段。

### 4.2 `SERVER_HOST` / `SERVER_PORT` 读取后基本未使用

证据：

- 顶部读取 `SERVER_HOST`、`SERVER_PORT`，但实际导出服务用的是 `PORT = process.env.PORT || 4001` 和 `hostname: '0.0.0.0'`。

影响：

- 配置文件里的 `server.port` 不会改变实际监听端口，容易造成部署误解。

位置：

- `packages/server/src/index.ts:16-18`
- `packages/server/src/index.ts:761-768`

建议：

- `PORT` 应优先使用 `config.server.port`。
- 如确实固定监听 `0.0.0.0:4001`，删除无效配置或仅作为 public URL 元数据。

### 4.3 `/api/slam/status` 在云服务器上检查本机 ROS，可能误导

证据：

- 该接口执行云服务器本机的 `tmux`、`pgrep slam_toolbox`、`ros2 topic info /map`。
- 真实 SLAM 在机器人 `192.168.1.58` 上运行。

影响：

- 云部署时该接口不能准确反映 Jetson SLAM 状态。
- 目前 UI 同时用 `/system/status` 和 `/api/slam/status`，本地 fallback 可能和机器人状态冲突。

位置：

- `packages/server/src/index.ts:713-739`
- `packages/client/src/App.tsx:111-117`

建议：

- 云部署中只以 `/system/status` 为准。
- `/api/slam/status` 改名为 `/api/local-sim/slam/status`，或通过 rosbridge/service 查询机器人状态。

### 4.4 MapCanvas 初始视图不自动 fit 地图

证据：

- 初始 `view` 是 `{ scale: 50, offsetX: 0, offsetY: 0 }`。
- 加载地图后没有根据地图尺寸和 origin 自动居中。

影响：

- 地图可能出现在屏幕外或只显示一角，用户误以为没有地图。

位置：

- `packages/client/src/components/MapCanvas.tsx:53`
- `packages/client/src/components/MapCanvas.tsx:229-239`

建议：

- 首次加载 `displayMapData` 后计算 fit-to-view。
- 提供“回到地图”按钮。

### 4.5 测试文件与当前中文 UI/隐藏控件脱节

证据：

- e2e 期望 `Layers`、`Map`、`Data Reception`、`Show Debug` 等英文文案。
- 当前 UI 使用中文 `图层`、`地图`、`显示调试`，并隐藏了 Data Reception。

影响：

- 即使安装了 Playwright 浏览器，现有 e2e 也大概率会因文案/控件不存在失败。
- 当前单元测试只测试内联数学表达式，没有直接覆盖真实导出函数或组件行为。

位置：

- `packages/client/tests/e2e.spec.ts`
- `packages/client/tests/utils.test.ts`
- `packages/client/src/components/LayerControl.tsx`

建议：

- 更新 e2e 到当前中文 UI，或给关键控件添加稳定 `aria-label` / `data-testid`。
- 为地图解析、导航 pose、连接重试、静态地图加载增加真实逻辑测试。

## 5. 逐文件备注

### 根目录 `package.json`

- 根入口清晰，但根 `dev` 是 local，server 包内 `dev` 是 cloud；团队协作时容易启动错 profile。
- 建议在 README 或脚本名上明确：根目录开发用 `bun dev`，云端服务用 systemd 的 `start:cloud`。

### `packages/server/src/config.ts`

- 轻量 parser 当前够用，但不是 YAML parser。
- `robot_config.cloud.yaml` 和用户当前机器人 IP 信息存在不一致，应确认。

### `packages/server/src/index.ts`

- API 功能集中在单文件，地图、媒体、Janus、Face、SLAM 状态混在一起，后续维护风险偏高。
- 地图接口需要路径安全和 PGM parser 修复。
- `/api/network` 需要 ESM 修复。
- `server.port` 配置未生效。
- CORS 全开 + 无鉴权，至少上传/删除地图、媒体控制接口应加保护。

### `packages/client/src/App.tsx`

- UI 主状态都集中在 App 内，建图/导航/媒体控制耦合较重。
- 导航启动失败、保存地图失败缺少显式 UI 错误。
- `selectedMap` 只用于构造机器人本地路径，没有用于画布静态地图显示。

### `packages/client/src/components/MapCanvas.tsx`

- 层次清楚，canvas 分层合理。
- 主要风险是静态地图未加载、初始视图不居中、目标朝向预览坐标系可能不一致。
- `selectedMap` 参数当前未被使用，应尽快处理，否则从调用方看起来像已支持静态地图。

### `packages/client/src/hooks/useRosConnection.ts`

- 自动重连思路合理。
- 手动 reconnect 可能被 stale ref 阻挡。
- React StrictMode 下 mount/unmount 会更频繁，现有双 cleanup effect 可以工作但有重复清理，建议简化成一个 effect。

### `packages/client/src/hooks/useSystemManager.ts`

- ROS service 调用封装简洁。
- 服务失败只写 state，不向调用方返回，导致 UI 很难准确反馈。
- `callService(..., 10)` 的第四个参数语义需要确认 roslib v2 是否为 timeout 秒；建议封装超时错误并统一显示。

### `packages/client/src/hooks/useNavigationTasks.ts`

- 能实现浏览器端简单巡航。
- 任务队列放在浏览器端，刷新/断线即丢失；更适合演示，不适合无人值守巡航。
- 未实际使用 `NavigateThroughPoses`。

### `packages/client/src/hooks/useRobotMedia.ts`

- 资源清理比较完整。
- 外部 CDN 是现场部署风险。

- 错误信息区域较小，长错误可能挤压布局；建议 title 或可展开日志。

#### `packages/client/src/components/MediaPanel.tsx`

作用：旧媒体控制面板，当前 App 已改用 `MediaViewport`。

潜在问题：

- 未使用组件仍会参与 TypeScript build；目前能过。
- 文案和布局与当前左下媒体控制重复，建议删除或标注 legacy，减少维护面。

#### `packages/client/src/components/ConnectionStatus.tsx`

作用：连接状态显示。

潜在问题：

- `error` 截断显示，只有 title 能看完整；移动端不方便查看。
- 连接中时重连按钮通常不显示，所以 `disabled={isConnecting}` 分支实际很少触发。

#### `packages/client/src/components/ui/*.tsx`

作用：Button/Badge/Card 基础组件。

潜在问题：

- Card 使用 `rounded-xl`，和项目设计约束中“卡片 8px 或更少”的偏好不一致，但不是功能 bug。
- 组件简单，未发现运行级 bug。

#### `packages/client/src/hooks/useRosConnection.ts`

作用：rosbridge WebSocket 连接和自动重连。

潜在问题：

- 手动 reconnect stale ref，见 3.3。
- 有两个 cleanup effect，开发态 StrictMode 下会重复关闭连接；目前 sessionId 防护能减少副作用，但代码可简化。

#### `packages/client/src/hooks/useRosMap.ts`

作用：订阅 `/map`。

潜在问题：

- `robotPose` state 在该 hook 内没有任何订阅来源，只暴露 `setRobotPose`；实际姿态主要由 `useRosTfTree` 提供。建议删除或整合。
- paused 时忽略新消息，但不会保留最后一帧刷新状态；符合暂停语义。

#### `packages/client/src/hooks/useRosTf.ts`

作用：订阅 `/tf` 和 `/tf_static`，解析 map 到目标 frame 的 2D pose。

潜在问题：

- 只从 `base_link`、`body`、`camera_init` 中选择机器人位姿，未包含 `base_footprint`。
- 每次 flush 都遍历全部 cache values 做 BFS；TF 边很多时性能会下降，但当前机器人规模可接受。
- 不按 timestamp 处理 TF，只使用最新覆盖值。

#### `packages/client/src/hooks/useRosScan.ts`

作用：轻量 LaserScan 订阅，供 MapCanvas 使用。

潜在问题：

- 不做点云转换，只保留 ranges；转换在 MapCanvas 做。
- 没有 compression；高频大 scan 经 rosbridge 可能压力较大。

#### `packages/client/src/hooks/useRosLaserScan.ts`

作用：另一个 LaserScan hook，转换成 Cartesian points。

潜在问题：

- 当前 MapCanvas 使用 `useRosScan`，这个 hook 看起来是旧实现。
- 使用 `compression: 'png'` 对 LaserScan 是否被 rosbridge 支持需要确认；LaserScan 不是 Image，可能无效。
- 每帧 console.log 点数，高频 scan 会污染控制台、影响性能。

#### `packages/client/src/hooks/useRosPath.ts`

作用：订阅 `/plan`，发布 `/goal_pose` 和 `/initialpose`。

潜在问题：

- `/plan` topic 在 Nav2 中可能是 `/plan`、`/global_plan` 或 planner server 私有话题；当前配置里的 `topics.global_plan` 没有被使用。
- goal publisher 当前基本被 `useNavigationTasks` 的 action 替代，但 fallback 仍存在。

#### `packages/client/src/hooks/useNavigationTasks.ts`

作用：Nav2 action 单点/多点/循环任务。

潜在问题：

- action 结果状态判断风险，见 3.4。
- 多点任务在浏览器端串行，见 3.5。
- `navigateThroughPosesAction` 配置未使用。

#### `packages/client/src/hooks/useSystemManager.ts`

作用：调用机器人 `/system/*` 服务。

潜在问题：

- 错误只写入状态，UI 展示不足，见 3.2。
- 状态 message 用 `mode|pid` 字符串协议，缺少 schema；如果后端 message 包含额外 `|` 会解析异常。

#### `packages/client/src/hooks/useKeyboardTeleop.ts`

作用：键盘发布 `motion_msgs/msg/MotionCtrl`。

潜在问题：

- 只发布 Diablo MotionCtrl，不再发布 `/cmd_vel`；如果机器人侧切换成标准 nav/teleop，这个 hook 不适用。
- A/D 语义注释为 rotation，但字段发到 `value.left`，具体机器人消息语义需要确认；可能是横移而非转向。
- keyup 只在 pressedKeysRef 为空时发送 stop；W+A 同时按住后松开 A 会继续发 W，行为合理。

#### `packages/client/src/hooks/useRobotMedia.ts`

作用：Janus streaming/audio/talkback 前端控制。

潜在问题：

- CDN 依赖，见 3.7。
- 页面加载时发现视频服务 active 会主动 stop，可能影响多客户端观看，见 4.5 之前的文件备注。
- preferred stream id 如果配置错误，不 fallback 到同类型 stream；`pickStream` 在 preferredId > 0 时只精确匹配。
- `stopTalkback()` 会停止所有匹配 forwarders；多客户端对讲时可能互相影响。

#### `packages/client/src/hooks/useFaceRecognition.ts`

作用：轮询 `/api/face/latest`。

潜在问题：

- 只有视频 connected 时 active，所以如果只想验证人脸服务状态但不开视频，不会轮询。
- `error` 返回但当前 App 未展示。

#### `packages/client/src/hooks/useSlamControl.ts`

作用：调用后端 `/api/slam/status` 和 `/api/maps`。

潜在问题：

- `/api/slam/status` 云端语义不准，见 4.3。
- `fetchMaps()` 没检查 `res.ok`，后端 500 也会尝试 json 并把 maps 置空。
- `deleteMap()` 不检查删除是否成功。

#### `packages/client/src/hooks/useMode.tsx`

作用：teleop/navigation 模式状态。

潜在问题：

- 切换模式不清理 `selectedMap`、`patrolPoints`、`navigationTaskMode`；跨模式回到导航会保留旧状态，可能符合便利性，也可能造成跨地图误操作。

#### `packages/client/src/hooks/useSubscriptionControl.ts`

潜在问题：

#### `packages/client/src/hooks/usePerformanceMonitor.tsx`

潜在问题：

#### `packages/client/src/lib/utils.ts`

作用：Tailwind class merge。

潜在问题：未发现。

作用：Tailwind base 和全局 body。

### 9.4 部署模板与运维工具

#### `deploy/nginx/webbot.conf`

作用：云端 HTTPS、静态前端、API、rosbridge、Janus demo 反代。

潜在问题：

- HTTP 访问 IP 会 301 到域名；如果证书/域名解析异常，IP 调试不方便。
- `/api/` 没有限制危险接口来源，依赖后端自保；但后端当前地图上传/删除无鉴权。
- `/rosbridge/` 长连接 timeout 足够长，但没有访问控制。

#### `deploy/systemd/webbot-server.service`

作用：云端 Bun 后端 systemd。

潜在问题：

- `User=root` 运行后端，地图文件和 sqlite 文件都会 root 拥有；建议降权到专用用户。
- `WorkingDirectory=/opt/webbot/packages/server` 导致 `MAPS_DIR` 是 `/opt/webbot/packages/server/maps`，这和本地开发可预期，但要纳入备份。

#### `deploy/sshd/99-webbot-tunnels.conf`

作用：云服务器 SSH 隧道保活设置。

潜在问题：

- 全局开启 `AllowTcpForwarding yes`，如果服务器上还有其他用户，需要配合 Match User 收窄。

#### `tools/ops-status.ts`

作用：云端和机器人服务状态探针。

潜在问题：

- 直接 SSH cloud host 和 jetson host，依赖本机 SSH key 与网络可达。
- cloud profile 下 jetson host 语义需澄清，见 2.3。
- `fetchJson` 对 HTTPS 云端域名没有使用，cloud probe 用 `http://${CLOUD_HOST}/api/health`，而 Nginx 会 301 到 HTTPS 域名；fetch 默认会跟随，但如果证书/域名不匹配可能受影响。

#### `tools/probe-rosbridge.ts`

作用：WebSocket rosbridge 可达性探针。

潜在问题：

- 默认 URL 固定 `wss://qiuhua.ying-guang.com/rosbridge/`，本地开发必须手动 `--url`。
- 只检查 WebSocket open，不检查 rosbridge protocol 是否响应。

## 10. 本次最建议立即修的 5 个点

1. `index.ts` 地图接口：地图名白名单、resolve 后校验目录、上传/删除加 token。

5. `useRosTf.ts`：加入 `base_footprint` 作为机器人位姿候选，并允许从 config 下发 frame 优先级。
